## 28.1 Claude Code 产品定位

### 28.1.1 产品定义

> **Claude Code** is an agentic coding tool that reads your codebase, edits files, runs commands, and integrates with your development tools.

**关键特性**:
- **Agentic**: 自主执行任务，不是被动问答
- **代码库理解**: 理解整个项目结构，不是单文件编辑
- **工具集成**: 与 git、终端、IDE 等开发工具深度集成
- **多表面支持**: Terminal、VS Code、JetBrains、Desktop、Web、iOS、Slack

### 28.1.2 目标用户与场景

| 用户类型 | 核心需求 | Claude Code 解决方案 |
|---------|---------|-------------------|
| **独立开发者** | 快速实现功能、修复 bug | 描述需求→自动实现 |
| **团队开发** | 代码审查、标准化 | CLAUDE.md 规范、Hooks 检查 |
| **DevOps** | 自动化运维任务 | 定时任务、CI/CD 集成 |
| **企业用户** | 安全、合规、可审计 | 数据隔离、审计日志 |

### 28.1.3 核心价值主张

**效率提升**:
- 自动化重复任务（写测试、修复 lint 错误、解决合并冲突）
- 多 Agent 并行执行，效率翻倍
- 定时任务自动化（晨间 PR 审查、夜间 CI 失败分析）

**质量保障**:
- CLAUDE.md 固化编码规范
- Hooks 机制自动检查
- 多 Agent 协作相互审查

**知识沉淀**:
- Auto Memory 自动学习项目知识
- 跨会话记忆，避免重复错误
- 团队共享 Skills 和规则

## 28.2 核心原理一：Context Engineering

### 28.2.1 核心思想

```
AI 的输出质量 = 模型能力 × 上下文质量

模型能力相同时，上下文质量决定输出质量
```

Claude Code 的整个架构围绕**如何提供高质量上下文**设计。

### 28.2.2 上下文架构

```
┌─────────────────────────────────────────────────────────────────┐
│                    Claude Code 上下文栈                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Layer 1: 项目结构索引                                           │
│  - 自动扫描目录结构                                              │
│  - 理解模块依赖关系                                              │
│  - 跨文件引用追踪                                                │
│                                                                 │
│  Layer 2: CLAUDE.md 规范层                                       │
│  - 组织级规范（/Library/Application Support/...）                 │
│  - 项目级规范（./CLAUDE.md）                                     │
│  - 用户级规范（~/.claude/CLAUDE.md）                             │
│  - 模块化规则（.claude/rules/*.md）                              │
│  - 路径特定规则（条件加载）                                      │
│                                                                 │
│  Layer 3: Auto Memory 经验层                                     │
│  - 构建命令                                                      │
│  - 调试经验                                                      │
│  - 项目特殊配置                                                  │
│  - 用户偏好（Claude 发现的）                                      │
│                                                                 │
│  Layer 4: 会话上下文                                             │
│  - 当前对话历史                                                  │
│  - 已执行操作                                                    │
│  - 待完成任务                                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**关键设计**:
- **分层加载**: 从通用到具体，优先级清晰
- **条件加载**: 路径特定规则只在相关文件打开时加载，节省 Token
- **自动学习**: Auto Memory 从工作中学习，无需手动维护

### 28.2.3 CLAUDE.md 分层规范

**文件位置（多层级）**:

| 范围 | 位置 | 用途 | 共享对象 |
|------|------|------|---------|
| **组织级** | `/Library/Application Support/ClaudeCode/CLAUDE.md` (macOS) | 组织规范、安全政策 | 组织内所有用户 |
| **项目级** | `./CLAUDE.md` 或 `./.claude/CLAUDE.md` | 团队共享规范 | 团队成员（通过版本控制） |
| **用户级** | `~/.claude/CLAUDE.md` | 个人偏好 | 仅自己（所有项目） |

**内容结构示例**:
```markdown
# CLAUDE.md

## 编码规范
- 使用 TypeScript，严格模式
- 函数不超过 50 行
- 所有公共 API 必须有文档

## 架构决策
- 使用 React + Vite
- 状态管理用 Zustand
- API 用 React Query

## 首选库
- 测试：Vitest + Testing Library
- 格式化：Prettier
- Lint：ESLint

## 审查清单
- [ ] 所有测试通过
- [ ] 无 ESLint 错误
- [ ] 类型检查通过
```

**最佳实践**:
- **大小**: 每个 CLAUDE.md 文件控制在 200 行以内
- **结构**: 使用 Markdown 标题和项目符号分组
- **具体性**: 写可验证的指令（"使用 2 空格缩进"而非"正确格式化代码"）
- **一致性**: 定期审查，移除过时或冲突的指令

### 28.2.4 .claude/rules/：模块化规则

**设计理念**:
```
将大型 CLAUDE.md 拆分为模块化规则文件，便于维护
```

**目录结构**:
```
your-project/
├── .claude/
│   ├── CLAUDE.md           # 主项目指令
│   └── rules/
│       ├── code-style.md   # 代码风格指南
│       ├── testing.md      # 测试规范
│       └── security.md     # 安全要求
```

**路径特定规则（Path-Specific Rules）**:
```markdown
---
paths:
  - "src/api/**/*.ts"
---

# API 开发规范

- 所有 API 端点必须包含输入验证
- 使用标准错误响应格式
- 包含 OpenAPI 文档注释
```

**优势**:
- **模块化**: 每个规则文件职责单一
- **条件加载**: 只在相关文件打开时加载，节省 Token
- **可维护**: 规则变更不影响其他规则

### 28.2.5 Auto Memory：自动学习

**设计理念**:
```
AI 应该从工作中学习，而不是每次都从头开始
```

**学习内容**:
- 构建命令
- 调试经验
- 项目特殊配置
- 用户偏好（Claude 发现的）

**存储位置**:
```
~/.claude/projects/<project>/memory/
```

**与 CLAUDE.md 的区别**:

| 特性 | CLAUDE.md | Auto Memory |
|------|-----------|-------------|
| **谁编写** | 用户手动编写 | Claude 自动学习 |
| **内容类型** | 指令和规则 | 学习心得和模式 |
| **作用范围** | 项目/用户/组织 | 每个工作树 |
| **加载方式** | 每次会话加载 | 前 200 行或 25KB |
| **用途** | 编码标准、工作流 | 构建命令、调试经验 |

## 28.3 核心原理二：Constraint Design

### 28.3.1 核心思想

```
AI 的可靠性 = 能力 - 自由度

自由度越低，可靠性越高
```

Claude Code 通过**多层约束**限制 AI 的自由度，让输出更可靠。

### 28.3.2 约束金字塔

```
┌─────────────────────────────────────────────────────────────────┐
│                  Claude Code 约束金字塔                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│         /\                                                      │
│        /  \                                                     │
│       /    \   Layer 5: Hooks（前后置钩子）                      │
│      /------\   - PreToolUse 验证                               │
│     /        \  - PostToolUse 检查                              │
│    /----------\  - 可阻止危险操作                                │
│   /            \                                                │
│  /--------------\  Layer 4: 工具权限控制                         │
│ /                \  - Sub-agent 工具限制                         │
│/------------------\  - MCP 工具权限                              │
│                    \                                            │
│ ---------------------\  Layer 3: 规范约束                        │
│                       \  - CLAUDE.md 规范                        │
│                        \  - 路径特定规则                          │
│                         \                                       │
│ -------------------------\  Layer 2: 记忆约束                    │
│                           \  - Auto Memory 经验                  │
│                            \                                     │
│ -----------------------------\  Layer 1: 模型约束                │
│                               \  - System Prompt                │
│                                \                                │
└─────────────────────────────────────────────────────────────────┘
```

**关键设计**:
- **多层约束**: 每层都有约束，层层把关
- **可执行约束**: Hooks 可以实际阻止危险操作
- **条件约束**: 路径特定规则只在相关文件上生效

### 28.3.3 Hooks 机制

**Hook 事件生命周期**:

| 事件 | 触发时机 | 用途 |
|------|---------|------|
| `SessionStart` | 会话开始 | 初始化、环境检查 |
| `UserPromptSubmit` | 用户提交提示 | 提示审查、增强 |
| `PreToolUse` | 工具调用前 | 验证、阻止危险操作 |
| `PermissionRequest` | 权限请求时 | 自动决策 |
| `PostToolUse` | 工具调用后 | 格式化、测试 |
| `PostToolUseFailure` | 工具调用失败 | 错误处理、重试 |
| `SubagentStart` | Sub-agent 启动 | 初始化 |
| `SubagentStop` | Sub-agent 停止 | 清理、汇总 |
| `Stop` | Claude 响应完成 | 后处理 |
| `SessionEnd` | 会话结束 | 清理、保存 |

**Hook 配置示例**:
```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/block-rm.sh"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "prettier --write"
          }
        ]
      }
    ]
  }
}
```

**Hook 脚本示例** (block-rm.sh):
```bash
#!/bin/bash
# .claude/hooks/block-rm.sh
COMMAND=$(jq -r '.tool_input.command')

if echo "$COMMAND" | grep -q 'rm -rf'; then
  jq -n '{
    hookSpecificOutput: {
      hookEventName: "PreToolUse",
      permissionDecision: "deny",
      permissionDecisionReason: "Destructive command blocked by hook"
    }
  }'
else
  exit 0  # 允许命令
fi
```

### 28.3.4 工具权限控制

**Sub-agent 工具限制**:
```markdown
---
name: code-reviewer
description: 审查代码质量和最佳实践
tools: Read, Glob, Grep
model: sonnet
---

你是一位代码审查员。当被调用时，分析代码并提供反馈。
```

**权限模式**:
- `default`: 默认权限（需要确认危险操作）
- `acceptEdits`: 自动接受编辑
- `dontAsk`: 不询问（全自动）
- `bypassPermissions`: 绕过权限（高风险）
- `plan`: 仅计划模式（只读）

## 28.4 多 Agent 协作系统

### 28.4.1 Sub-agents 设计

**设计理念**:
```
复杂任务分解给多个 Agent 并行执行，主 Agent 协调
```

**内置 Sub-agents**:

| Agent | 模型 | 工具 | 用途 |
|-------|------|------|------|
| **Explore** | Haiku | 只读工具 | 代码库探索、文件发现 |
| **Plan** | 继承 | 只读工具 | Plan mode 研究 |
| **General-purpose** | 继承 | 所有工具 | 复杂多步任务 |
| **Bash** | 继承 | Bash | 终端命令执行 |

**工作流程**:
```
用户请求
    ↓
主 Agent（任务规划）
    ↓
┌───────────┬───────────┬───────────┐
│ Sub-Agent1│ Sub-Agent2│ Sub-Agent3│
│ 前端修改  │ 后端修改  │ 测试编写  │
└───────────┴───────────┴───────────┘
    ↓
主 Agent（结果合并）
    ↓
最终输出
```

### 28.4.2 自定义 Sub-agent 配置

**完整配置示例**:
```markdown
---
name: code-reviewer
description: 审查代码质量和最佳实践
tools: Read, Glob, Grep
model: sonnet
permissionMode: default
maxTurns: 10
skills: [code-review-skill]
memory: user
---

你是一位高级代码审查员。当被调用时，分析代码并提供具体、可操作的反馈。

## 审查要点
1. 代码规范符合性
2. 潜在安全问题
3. 性能优化建议
4. 测试覆盖率
```

**配置选项**:

| 字段 | 必填 | 说明 |
|------|------|------|
| `name` | 是 | 唯一标识符 |
| `description` | 是 | 何时委托给此 Sub-agent |
| `tools` | 否 | 可用工具列表 |
| `disallowedTools` | 否 | 禁止的工具 |
| `model` | 否 | 使用的模型 |
| `permissionMode` | 否 | 权限模式 |
| `maxTurns` | 否 | 最大执行轮数 |
| `skills` | 否 | 加载的技能 |
| `memory` | 否 | 持久记忆范围 |

## 28.5 MCP 协议：工具集成标准

### 28.5.1 MCP 架构

**设计理念**:
```
MCP 是连接 AI 与外部工具的开放标准协议
```

**支持的传输方式**:
- **HTTP**（推荐）: 远程 HTTP 服务器
- **SSE**（已弃用）: Server-Sent Events
- **Stdio**: 本地进程

**配置示例**:
```bash
# HTTP 远程服务器
claude mcp add --transport http notion https://mcp.notion.com/mcp

# Stdio 本地服务器
claude mcp add --transport stdio --env AIRTABLE_API_KEY=YOUR_KEY airtable \
  -- npx -y airtable-mcp-server
```

### 28.5.2 MCP 工具命名规范

**命名模式**:
```
mcp__<server>__<tool>

示例:
- mcp__memory__create_entities
- mcp__filesystem__read_file
- mcp__github__search_repositories
```

**Hook 匹配**:
```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "mcp__memory__.*",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'Memory operation' >> ~/mcp-operations.log"
          }
        ]
      }
    ]
  }
}
```

## 28.8 Harness Engineering 深度思考

### 28.8.1 核心类比：仓库是 Agent 的操作系统

> **CPU 很强大，但没有操作系统它就是一块高速却无方向的芯片**
> ——不知道硬盘在哪、网络协议是什么、哪些内存地址可以写。
>
> **LLM 也一样**。它推理能力很强，但它不知道：
> - 你的 `internal/types/` 不能 import `internal/config/`
> - 新文件该放哪个目录
> - 哪些依赖是允许的
>
> **Harness 就是给它装的"操作系统"**。

### 28.8.2 四条关键原则

#### 原则一：仓库是唯一的事实来源

**核心思想**:
```
Aone 里的讨论、钉钉会议上的口头约定、架构师脑子里的蓝图
这些对 Agent 来说都不存在。

不在仓库里，Agent 就看不见；
看不见就会违反。
```

**实践方法**:
- ✅ 把一切编码到仓库中（架构决策、层级约束、命名规范）
- ❌ 不要写在 Wiki 里、发在群里、口头约定
- ✅ 作为版本化的文件提交到 Git
- ✅ 知识跟着代码一起走

**为什么？**
```
Agent 只能看到:
├── Git 仓库内的文件
├── 当前会话的对话历史
└── 工具返回的结果

Agent 看不到:
├── Aone/钉钉/Slack 讨论
├── 会议上的口头约定
├── 架构师脑子里的蓝图
└── Wiki 上的文档（除非明确读取）
```

#### 原则二：AGENTS.md 是地图，不是手册

**核心思想**:
```
当一切都重要时，什么都不重要。

500 行的指令文件挤占了 Agent 宝贵的上下文窗口，
留给实际任务的空间反而少了。

AGENTS.md 应该是地图，不是手册
——控制在 ~100 行，只做索引和指路。
```

**错误做法**:
```markdown
# AGENTS.md (500 行)
## 编码规范 (100 行)
## 架构决策 (100 行)
## 命名规范 (100 行)
## ... (更多)
```

**正确做法**:
```markdown
# AGENTS.md (~100 行)

## 快速导航
- 项目结构：[docs/structure.md](docs/structure.md)
- 架构决策：[docs/architecture.md](docs/architecture.md)
- 编码规范：[docs/coding-standards.md](docs/coding-standards.md)

## 关键约束
- internal/types/ 不能 import 任何 internal/ 包
- 所有公共 API 必须有 JSDoc
- 测试覆盖率 > 80%
```

**详细内容放在 docs/**:
```
docs/
├── structure.md          # 项目结构详情
├── architecture.md       # 架构决策详情
└── coding-standards.md   # 编码规范详情
```

**优势**:
- 按需加载，节省 Token
- 不容易腐烂
- 容易维护
- 查找快速

#### 原则三：约束的粒度

**核心思想**:
```
Harness 不规定"你必须用这个设计模式"或"函数必须这样写"，
它只管架构边界。

规则就一条：高层可以 import 低层，反过来不行。
在这个边界之内怎么实现，随便。

跟管理大型平台团队一样：中心化约束，本地自治。
```

**层级编号系统**:
```
Layer 0: 类型定义（不 import 任何内部包）
    ↓
Layer 1-2: 工具函数和配置（只依赖 Layer 0）
    ↓
Layer 3: 业务逻辑（可 import Layer 0-2）
    ↓
Layer 4+: HTTP handler 和 CLI 命令（可 import Layer 0-3）
```

**实施方法**:
```markdown
# .claude/rules/layering.md

## 层级定义
- Layer 0: src/types/（无依赖）
- Layer 1-2: src/utils/, src/config/（只依赖 Layer 0）
- Layer 3: src/logic/（可 import Layer 0-2）
- Layer 4+: src/http/, src/cli/（可 import Layer 0-3）

## 依赖规则
- 高层可以 import 低层
- 低层不能 import 高层
- 同层之间不能循环依赖
```

#### 原则四：人的角色转变

**核心思想**:
```
以前是人写代码、AI 辅助补全；
现在反过来了——

人设计系统（架构、约束、验证规则），
Agent 在系统内执行。

人的价值从"写出正确的代码"
变成了"设计出让 Agent 能可靠产出正确代码的环境"。

你不再需要自己拧每一颗螺丝，
但你得确保流水线是对的。
```

**角色对比**:

| 维度 | 传统模式 | Agent 模式 |
|------|---------|-----------|
| **人的工作** | 写代码 | 设计系统 |
| **AI 的工作** | 辅助补全 | 执行任务 |
| **价值所在** | 写出正确的代码 | 设计可靠的环境 |
| **关注点** | 实现细节 | 架构约束 |
| **产出** | 代码 | 系统 + 约束 |

**新技能要求**:
- ✅ 系统设计能力（架构设计、约束设计、验证规则设计）
- ❌ 不再是手写每一行代码、记忆所有规范、手动检查每一处依赖

### 28.8.4 落地实施：双引擎架构

**核心架构**:
```
Harness 靠两个引擎协作：

harness-creator ← 负责搭建
    ↓
harness-executor ← 负责执行
```

**harness-creator（搭建引擎）**:
- 分析代码库
- 生成基础设施（文档、lint 脚本、目录结构）
- 项目健康度评分（0-100 分）

**harness-executor（执行引擎）**:
- 在基础设施中执行开发任务
- 遵循约束规则
- 生成执行计划
- 验证结果

**协作关系**:
```
executor 启动
    ↓
检查 AGENTS.md 在不在？
    │
    ├─ 在 → 直接开始执行
    │
    └─ 不在 → 自动喊 creator 来搭建
                ↓
              creator 搭建完成
                ↓
              executor 继续干活
```

**关键设计**:
```
任何项目都能直接用 executor 起步
→ 无需手动配置
→ 自动检测和搭建
→ 渐进式完善
```

### 28.8.5 项目健康度评分

**评分维度**:

| 维度 | 说明 | 权重 |
|------|------|------|
| **文档覆盖率** | 关键文档是否齐全 | 30% |
| **lint 规则覆盖率** | 代码规则是否完整 | 25% |
| **测试覆盖率** | 测试是否充分 | 25% |
| **架构清晰度** | 层级约束是否明确 | 20% |

**评分等级**:

| 分数 | 等级 | 状态 | 行动 |
|------|------|------|------|
| **0-20 分** | 裸奔状态 | 基本无约束 | 从零搭建全套 |
| **21-70 分** | 有基础 | 有缺口 | 针对性补充 |
| **71-100 分** | 健康 | 比较完善 | 微调就行 |

**实施步骤**:
```
0-20 分项目:
1. creator 审计项目
2. 生成基础设施（AGENTS.md、docs/、scripts/、harness/）
3. 评分从 0 分 → 70+ 分
4. executor 开始执行

21-70 分项目:
1. creator 审计项目，识别缺口
2. 针对性补充（缺失文档、脚本、规则）
3. 评分从 50 分 → 80+ 分

71-100 分项目:
1. creator 审计项目，识别优化点
2. 微调（更新文档、优化脚本、补充经验）
3. 评分保持 80+ 分
```

### 28.8.6 executor 工作流

**完整流程**:
```
1. 检测环境 → 检查 AGENTS.md、docs/、scripts/
2. 加载上下文 → 读取 AGENTS.md、相关 docs/
3. 制定计划 → 创建执行计划文件 (exec-plans/active/)
4. 人类批准 ← 关键步骤！（不是走过场）
5. 执行 → 按计划执行，记录轨迹 (harness/trace/)
6. 验证 → 运行验证脚本 (scripts/validate.py)
7. 完成 → 移动计划到 completed/，存储经验 (harness/memory/)
```

**执行计划文件**:
```markdown
# 执行计划：实现用户登录功能

## 任务目标
- 实现邮箱 + 密码登录
- 返回 JWT token
- 添加单元测试

## 影响范围
- 新增文件：src/auth/login.ts, src/auth/login.test.ts
- 修改文件：src/routes.ts, src/types.ts
- 不影响：其他模块

## 分阶段步骤
### 阶段 1: 类型定义
### 阶段 2: 实现登录逻辑
### 阶段 3: 集成路由

## 验证方式
- 单元测试
- 集成测试
- 手动测试

## 回退策略
- 如果失败：回滚 git 变更
```

**关键设计**:
```
"人类批准"这一步不是走过场——
对于非简单任务，executor 会创建执行计划文件。

你扫一眼就知道 Agent 打算怎么干，
觉得不对可以直接改方向。
```

## 28.9 核心原则：上下文是最贵的资源

### 28.9.1 核心论点

> **如果这篇文章你只记住一件事，记这个：**
> **中等复杂度以上的任务，协调者绝不写代码。**

### 28.9.2 为什么？

**上下文窗口的限制**:
```
LLM 的上下文窗口是有限的。

Agent 执行长任务时，窗口里慢慢堆起来的是：
├── 代码 diff
├── 编译错误
├── 测试输出
├── lint 报告
├── 重试记录
└── ...

到第 40 次 tool call：
→ 早期的关键信息已经被压缩或丢弃了
→ Agent 开始"忘记"最初的架构决策
→ 做出跟前面矛盾的改动
```

**恶性循环**:
```
任务越复杂
    ↓
消耗的上下文越多
    ↓
关键信息被压缩
    ↓
决策质量下降
    ↓
产生更多错误
    ↓
需要更多修复
    ↓
消耗更多上下文
    ↓
（恶性循环）

到第 60 次 tool call：
→ Agent 可能已经忘了最初的任务目标是什么了
→ 你让它加个用户认证功能
→ 它却在那儿纠结一个完全不相关的 import 路径问题
```

### 28.9.3 解法：两层架构

**协调者（Coordinator）+ 执行者（子代理）**:
```
┌─────────────────────────────────────────────────────────────────┐
│                    两层架构                                     │
├─────────────────────────────────────────────────────────────────┤
│  协调者（Coordinator）                                          │
│  ├── 职责：规划、委派、汇总                                     │
│  ├── 约束：一行代码都不碰                                       │
│  └── 上下文：保留摘要，不保留详细执行过程                       │
│                                                                 │
│  执行者（子代理）                                                │
│  ├── 职责：执行具体任务                                         │
│  ├── 特点：每次从干净的上下文开始                               │
│  ├── prompt：精确的 prompt，包含它需要知道的一切                │
│  └── 生命周期：干完就释放                                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**工作流程**:
```
协调者接收任务
    ↓
协调者规划
    ↓
协调者委派给子代理
    ↓
子代理从干净的上下文开始
    ↓
子代理执行任务
    ↓
子代理返回结果
    ↓
协调者汇总
    ↓
子代理释放（上下文清空）
    ↓
协调者只保留摘要
```

**关键设计**:
```
子代理不知道之前发生了什么，
但它拿到的 prompt 里包含了它需要知道的一切。

任务完成后：
- 详细上下文被丢掉
- 协调者只保留一段摘要
- 信息经过了压缩和筛选
- 不是无差别地堆在上下文里
```

### 28.9.4 最常见的陷阱

**"只是快速改一下"**:
```
破坏这个原则最常见的方式是"只是快速改一下"。

协调者发现一个小问题，心想：
"这么简单的东西不用启子代理，我直接改了"

一次编辑变成了五次（因为牵扯出其他地方）
五次变成二十次
上下文就被消耗殆尽了
```

**为什么这是陷阱？**
```
这个陷阱看起来很合理——
"就改一行而已嘛"

但你低估了代码的牵连性。

改一行代码：
├── 可能牵扯出 5 个相关文件
├── 每个文件又有依赖
├── 依赖又有依赖
└── ...

一次编辑 → 五次编辑 → 二十次编辑 → 上下文耗尽
```

**如何避免？**
```
如果你发现协调者正在用 Edit 或 Write 工具修改源代码，
立刻停下来，启动子代理。

没有例外。
```

### 28.9.5 不同复杂度的执行方式

**快速判断方法**:

| 任务类型 | 判断标准 | 执行方式 |
|---------|---------|---------|
| **简单任务** | 能用一句话描述且不包含"和"字 | 协调者直接做 |
| **中等任务** | 需要清单来跟踪改了哪些地方 | 委派给子代理 |
| **复杂任务** | 需要做设计决策和权衡 | 委派 + 隔离（Git Worktree） |

**示例**:
```
简单任务（协调者直接做）:
✅ 改个 typo
✅ 加行日志
✅ 更新注释

中等任务（委派给子代理）:
✅ 多文件一致性修改
✅ 重命名变量（跨文件）
✅ 添加单元测试

复杂任务（委派 + 隔离）:
✅ 重构
✅ 新模块
✅ 结构性变更
```

### 28.9.6 Git Worktree 隔离

**复杂任务执行**:
```
1. 创建 Git Worktree（仓库临时副本）
2. 子代理在 Worktree 中执行
3. 验证通过
   ├─ 成功 → 合并到主分支
   └─ 失败 → 丢掉 Worktree，不污染主分支
```

**好处**:
- ✅ 不污染主分支
- ✅ 失败无成本
- ✅ 可并行执行多个任务

### 28.9.7 模型选择策略

**核心洞察**:
```
委派子代理时还有一个容易被忽略的杠杆：
不是所有任务都需要用最强的模型。

一个"重命名变量"的任务和一个"重构认证模块"的任务，
对模型能力的要求完全不同。

前者要的是快和便宜
后者要的是深度推理

如果所有子代理都用同一个顶级模型，
既浪费钱又浪费时间。
```

**模型选择矩阵**:

| 任务类型 | 典型任务 | 推荐模型 | 选择理由 |
|---------|---------|---------|---------|
| **快速执行类** | 改 typo、简单重命名 | Claude Haiku | 响应快、成本低 |
| **深度推理类** | 复杂重构、架构级变更 | GPT-5.3 Codex / Claude Opus | 质量远比速度重要 |
| **代码检索类** | 在大型代码库中定位相关文件 | Gemini 3 Flash | 速度第一 |

**协调者委派示例**:
```markdown
协调者规划：
├── 任务 1: 改 typo（快速执行类）
│   └── 委派给：Claude Haiku
│
├── 任务 2: 重构认证模块（深度推理类）
│   └── 委派给：Claude Opus
│       └── 在 Git Worktree 里隔离执行
│
└── 任务 3: 定位相关文件（代码检索类）
    └── 委派给：Gemini 3 Flash
```

**好处**:
- ✅ 节省成本（简单任务不用顶级模型）
- ✅ 提升速度（简单任务快速完成）
- ✅ 保证质量（复杂任务用顶级模型）
- ✅ 资源优化（不同任务用不同模型）

### 28.9.8 实施建议

**协调者约束清单**:
```
协调者绝不：
❌ 用 Edit 工具修改源代码
❌ 用 Write 工具创建代码文件
❌ 直接执行代码
❌ 直接运行测试

协调者只做：
✅ 规划任务
✅ 委派给子代理
✅ 汇总结果
✅ 保留摘要
```

**子代理生命周期**:
```
子代理创建
    ↓
接收精确 prompt（包含需要知道的一切）
    ↓
从干净的上下文开始执行
    ↓
完成任务
    ↓
返回结果
    ↓
释放（上下文清空）
```

### 28.9.9 多模型协作与交叉 Review

**多模型协作工作流**:
```
一个中等复杂度的功能开发，可能会同时调度三四个不同模型的子代理：

协调者（中等模型）
└── 只做调度，不写代码
     ↓
├─→ 子代理 1: Flash 检索相关代码
├─→ 子代理 2: Opus 做核心实现
├─→ 子代理 3: Codex 做交叉 review
     ↓
汇总结果

总成本可以降低 60-70%
复杂任务的质量不打折扣
```

**为什么能降低成本？**
```
协调者本身用中等模型就行
→ 它不写代码，只做调度

子代理根据任务性质选择模型
→ 快速执行类：轻量模型（快 + 便宜）
→ 深度推理类：顶级模型（质量优先）
→ 代码检索类：速度第一

不是所有任务都需要用最强的模型
```

**交叉 Review 的价值**:
```
机械化验证（lint、test、verify）能抓到规则层面的问题，
但抓不到逻辑合理性——

代码编译通过、测试全绿、架构合规，
但实现思路可能有隐患：
├── 竞态条件
├── 边界遗漏
├── 不必要的复杂度
├── 命名让人看不懂
└── ...

这些东西只有"另一双眼睛"才能发现。
```

**关键词：不同模型**:
```
如果用同一个模型既写代码又 review，
它容易对自己的产出"视而不见"——
类似于人写完文章自己检查总觉得没问题。

换一个架构和训练数据都不同的模型来 review，
思维盲区的重叠就小得多。
```

**交叉 Review 流程**:
```
1. 编码子代理完成编码
   └── 提交代码变更

2. 机械化验证
   ├── lint 检查
   ├── 测试运行
   └── verify 验证

3. 交叉 Review ← 关键步骤！
   ├── Review 子代理拿到：
   │   ├── 变更的文件 diff
   │   ├── 任务描述
   │   └── 架构文档
   │
   ├── 检查内容：
   │   ├── 逻辑正确性
   │   ├── 边界条件
   │   ├── 命名清晰度
   │   └── 潜在性能问题
   │
   └── 输出结果：
       ├── "通过" → 进入下一步
       └── "需修改"（附具体建议）→ 回到编码子代理

4. 修复（如果需要）
   ├── 编码子代理根据 review 建议修复
   └── 再过一轮 review

5. 协调者接受结果
   └── 汇总、交付
```

**多模型协作示例**:
```markdown
任务：实现用户认证模块

协调者（Claude Sonnet - 中等模型）:
├── 职责：规划、委派、汇总
└── 成本：中等

子代理 1: Gemini 3 Flash（代码检索类）
├── 任务：检索相关代码
├── 职责：在大型代码库中定位相关文件
└── 成本：低（速度第一）

子代理 2: Claude Opus（深度推理类）
├── 任务：核心实现
├── 职责：实现用户认证逻辑
└── 成本：高（质量优先）

子代理 3: GPT-5.3 Codex（交叉 review）
├── 任务：交叉 review
├── 职责：检查逻辑正确性、边界条件、命名清晰度
└── 成本：高（质量优先）

总成本：比全部用 Opus 降低 60-70%
质量：不打折扣
```

**关键设计原则**:
1. **协调者不写代码**: 只做调度，用中等模型
2. **模型选择根据任务性质**: 快速/深度/检索用不同模型
3. **交叉 Review 用不同模型**: 避免思维盲区重叠
4. **Review 嵌入在流程中**: 机械验证之后、协调者接受之前

### 28.9.10 Review 精细化与 Harness 自我进化

**何时需要交叉 Review？**:
```
不是所有任务都需要交叉 review。

简单修改直接过机械验证就够了：
├── 改 typo
├── 加行日志
├── 简单重命名
└── ...

但涉及以下情况，加一轮是值得的：
├── 核心业务逻辑
├── 安全相关代码
├── 影响面较广的重构
└── ...
```

**成本分析**:
```
review 子代理只需要读 diff 和文档，
一次 review 的成本大约是编码成本的 10-20%。

编码子代理：读完整文件 + 写代码
review 子代理：读 diff + 读文档

成本对比：
编码：$1.00
review: $0.10-0.20 (10-20%)

性价比：
花 10-20% 的额外成本，
发现潜在的逻辑问题、边界遗漏、性能隐患。
```

**Harness 自我进化**:
```
review 中发现的问题会被记录到 harness/trace/ 里。

如果某类问题反复出现，
说明应该把它编码成新的 lint 规则——
把 review 中的"软知识"逐渐转化成验证管道中的"硬规则"。

这也是 Harness 自我进化的一部分。
```

**进化流程**:
```
1. Review 发现问题 → 记录到 harness/trace/
2. 问题分类 → 偶发/反复出现
3. 反复出现 → 编码成 lint 规则
4. 验证新规则 → 检查现有代码
5. 持续运行 → 新代码自动检查，问题不再出现
```

### 28.9.11 检查点机制

**为什么需要检查点？**:
```
中等以上的任务还会设检查点。

每完成一个阶段、跑过验证就存档，
包括已有的架构决策。

任务中断了，下一个 Agent 能从检查点恢复，
不会做出跟前面矛盾的选择。

检查点携带架构决策这一点很重要——
没有它，新 Agent 可能走一条完全不同的路，
引入微妙的不一致。
```

**检查点内容**:
```markdown
# 检查点：阶段 1 完成

## 基本信息
- 任务 ID: auth-module-implementation
- 阶段：1/3
- 时间：2026-03-27 23:00
- Agent: Claude Opus

## 架构决策 ← 关键！
- 决策 1: 使用 JWT 作为 token 格式
- 决策 2: token 有效期 24 小时
- 决策 3: 使用 bcrypt 加密密码
- 决策 4: 认证模块放在 src/auth/

## 已完成工作
## 变更文件
## 验证结果
## 下一步计划
```

**检查点时机**:
```
中等以上任务:
├── 每完成一个阶段
├── 跑过验证后
├── 存档检查点
└── 继续下一阶段

简单任务:
└── 不需要检查点（直接完成）
```

**检查点存储**:
```
harness/
└── checkpoints/
    ├── auth-module-implementation/
    │   ├── checkpoint-phase-1.md
    │   ├── checkpoint-phase-2.md
    │   └── checkpoint-phase-3.md
    └── other-task/
        └── ...
```

**恢复流程**:
```
1. 新 Agent 启动
   └── 检查 harness/checkpoints/{task-id}/

2. 找到最新检查点
   └── checkpoint-phase-2.md

3. 读取检查点
   ├── 了解架构决策
   ├── 了解已完成工作
   └── 了解下一步计划

4. 从检查点恢复
   └── 继续阶段 3
```

**好处**:
- ✅ 任务中断后可恢复
- ✅ 保持一致性（架构决策不丢失）
- ✅ 新 Agent 不会做出矛盾选择
- ✅ 多 Agent 协作更顺畅

### 28.9.12 Harness 自我进化与轨迹编译

**核心洞察**:
```
想想看：每次 Agent 犯错都是一个信号。

如果同一类错误反复出现，
说明 Harness 本身有缺口——

├── 要么某个包没被 linter 覆盖
├── 要么某条规则的错误信息写得不够清楚
└── 要么某个场景缺少文档

与其靠人去发现这些缺口，
不如让系统自己分析。
```

**自我进化闭环**:
```
┌─────────────────────────────────────────────────────────────────┐
│                    Harness 自我进化闭环                          │
├─────────────────────────────────────────────────────────────────┤
│  1. Agent 执行 → 2. 验证抓到问题 → 3. Critic 分析模式           │
│      ↓                                          ↓               │
│  5. 下一个 Agent 受益 ← 4. Refiner 更新规则                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**示例场景**:
```
场景：internal/cache 被 7 次违规 import

1. Agent 执行: 7 个不同的 Agent 都 import 了 internal/cache
2. 验证抓到问题：linter 抓到违规 import
3. Critic 分析模式：发现 internal/cache 没被加入层级映射表
4. Refiner 更新规则：把 internal/cache 加入 Layer 1
5. 下一个 Agent 受益：同样的错误不再出现
```

### 28.9.13 Harness 的三种记忆

**1. 情景记忆（Episodic Memory）**:
```
记录具体事件和教训。

比如：
"macOS 下 /var 是 /private/var 的符号链接，
会导致工作区路径比较失败"

价值：这类教训可能只要 10 秒加载，
      却能省下一整个重试循环。
```

**2. 程序记忆（Procedural Memory）**:
```
记录成功的操作步骤。

比如：
"添加 API 端点的标准五步流程，成功率 90%"

价值：新的子代理执行同类任务时会先查这里。
      直接复用成功流程，避免重新探索。
```

**3. 失败记忆（Failure Memory）**:
```
专门供 Critic 分析用。

记录所有验证失败的详细信息。

价值：Critic 分析失败模式，发现 Harness 缺口。
```

**记忆查询**:
```
每次任务开始时，executor 会查询相关记忆。

1. 查询情景记忆 → 有没有相关的教训？
2. 查询程序记忆 → 有没有成功的流程？
3. 查询失败记忆 → 有没有要避免的错误？

这些记忆的价值随着项目演进不断累积。
```

### 28.9.14 轨迹编译

**核心概念**:
```
当同一类任务被成功执行了三次以上，
而且步骤高度一致——

比如"添加 API 端点"每次都是：
├── 创建类型文件
├── 写服务方法
├── 加 handler
├── 注册路由
└── 写测试

这个模式就可以被"编译"成一个确定性脚本，
像 make add-endpoint NAME=foo 这样。

以后再做同类任务直接跑脚本，
LLM 都省了。

脚本失败了再回退到 Agent 模式就好。
```

**编译条件**:
```
触发条件:
├── 成功次数 ≥ 3
├── 步骤一致性 ≥ 90%
├── 验证通过率 = 100%
└── Review 通过率 = 100%
```

**编译流程**:
```
1. 模式检测 → Critic 分析成功轨迹
2. 提取步骤 → 提取共同步骤，参数化变量
3. 生成脚本 → 编写确定性脚本
4. 验证脚本 → 运行脚本测试
5. 注册脚本 → 添加到 scripts/
6. 持续使用 → 同类任务优先用脚本，失败回退到 Agent
```

**示例：添加 API 端点**:
```bash
# 编译后的脚本
./scripts/add-endpoint.sh foo

# 使用方式
make add-endpoint NAME=foo

# 脚本失败了再回退到 Agent
./scripts/add-endpoint.sh foo || claude "添加 API 端点 foo"
```

### 28.9.15 棘轮效应

**核心洞察**:
```
这是个棘轮效应：
每一个被编译的成功模式都变成了永久基础设施，
Agent 被释放去处理真正需要创造力的新问题。
整个系统的运行成本越来越低，能力越来越强。
```

**进化过程**:
```
第一轮:
├── 100% 任务用 Agent
└── 成本高，质量波动

第二轮:
├── 70% 任务用 Agent
├── 30% 任务用脚本（已编译）
└── 成本降低，质量稳定

第三轮:
├── 40% 任务用 Agent
├── 60% 任务用脚本（已编译）
└── 成本更低，质量更稳定

...

第 N 轮:
├── 10% 任务用 Agent（真正需要创造力的）
├── 90% 任务用脚本（已编译）
└── 成本最低，质量最稳定
```

**好处**:
- ✅ 运行成本越来越低
- ✅ 能力越来越强
- ✅ Agent 处理真正需要创造力的问题
- ✅ 重复性工作自动化
- ✅ 知识沉淀为基础设施

### 28.9.16 朴素方法的两个失败模式

**失败模式一：上下文焦虑（Context Anxiety）**
```
当对话历史越来越长，模型会开始"心慌"——
它觉得上下文快满了，于是开始草草收工。

这不是 bug，是模型在训练数据中学到的模式：
长对话到后面总该收尾了。

解法：上下文重置
不是压缩历史，而是直接新开一个 Agent，
把前一个 Agent 的成果通过结构化文件传递过去。

新 Agent 拿到的是：
├── 干净的上下文窗口
└── 清晰的任务接力棒

这和"压缩（Compaction）"是不同的：
- Compaction：在同一个对话里把早期内容做摘要
- Reset：彻底换一个 Agent，给它一个全新的"第一天上班"的心态

Anthropic 的发现：
Claude Sonnet 4.5 的上下文焦虑严重到压缩根本不够用，
必须做完整的上下文重置，这成了 Harness 设计的刚需。
```

**失败模式二：自我评价偏差（Self-Evaluation Bias）**
```
让 AI 评价自己的作品，它基本上永远觉得"挺好的"。
就像让厨师评价自己做的菜——能差到哪去？

解法：GAN 式生成 - 评价分离
把生成和评价拆成两个独立的 Agent。

生成器负责做，评价器负责挑刺。
评价器有自己独立的上下文，
不会被生成过程中的推理链"洗脑"。

关键的一点是：
调教一个独立的评价器使其变得严格，
比让生成器对自己的作品保持批判，要容易得多。

一旦外部的严格反馈存在了，
生成器就有了具体的改进方向。
```

### 28.9.17 三 Agent 架构

**Anthropic 最终落地的架构**:
```
┌───────────┐      ┌───────────┐      ┌───────────┐
│ Planner   │─────▶│ Generator │─────▶│ Evaluator │
│ 规划器    │      │ 生成器    │◀─────│ 评价器    │
│           │      │           │      │           │
│ 1 句话    │      │ 按 Sprint  │      │ Playwright│
│ →完整产品 │      │ 逐个实现  │      │ 实际操作  │
│ 规格书    │      │ 自带 git  │      │ 找真 bug  │
└───────────┘      └───────────┘      └───────────┘
```

**1. Planner（规划器）**:
```
把用户的一句话扩展成完整的产品规格书。

关键点：
它只管"做什么"，不管"怎么做"——
避免规划层面的错误级联到实现层。

成本极低但价值极高（$0.46，4.7 分钟）。
```

**2. Generator（生成器）**:
```
按 Sprint 逐个实现功能。

每个 Sprint 前和 Evaluator 协商一份"验收合同"，
明确"做完长什么样"。

成本大头（$71-36/轮，1-2 小时/轮）。
```

**3. Evaluator（评价器）**:
```
用 Playwright MCP 实际操作应用，像真实用户一样点击、输入、验证。

不是看代码，是看运行效果。

成本低但 ROI 高（$3-4/轮，10 分钟/轮）。
$3 的 QA 捞回了 $70 的 Build 差点交付的半成品。
```

### 28.9.18 Sprint 合同与 Evaluator 驯服

**Sprint 合同**:
```
产品规格是高层次的（比如"用户可以编辑关卡"），
而测试需要具体的可检验行为（比如"矩形填充工具支持点击拖拽填充"）。

Sprint 合同就是这个中间桥梁。
一个 Sprint 可以有 20-30 条具体的验收标准。

协商流程:
1. Generator 提出"我打算做什么、怎么验证算完成"
2. Evaluator 审查这个提案是否合理
3. 两边迭代到达成一致
4. Generator 才开始编码

通信方式很朴素——通过文件读写。
```

**Evaluator 的驯服**:
```
开箱即用的 Claude 是一个"很差的 QA Agent"。

典型模式：
Evaluator 确实发现了问题，
然后在日志里写了一段分析，
接着话锋一转——"不过这个问题不算严重"——
然后通过了。

驯服过程:
1. 跑一轮 Harness → 拿到 Evaluator 的完整日志
2. 人工逐条审阅，找到 Evaluator 的判断与人类判断不一致的地方
3. 更新 Evaluator 的提示词，让它在这些地方变得更严格
4. 再跑一轮，重复

经过好几轮这样的校准，Evaluator 才变得靠谱。
```

**四个评价维度**:
```
在前端设计任务中，Anthropic 设计了四个评价维度：

| 维度 | 评价标准 | 权重 |
|------|---------|------|
| 设计质量 | 颜色、排版、布局、图像是否创造了独特氛围 | 高 |
| 原创性 | 有没有定制化的设计决策 | 高 |
| 工艺 | 排版层次、间距一致性、色彩和谐度 | 中 |
| 功能性 | 用户能不能理解界面、完成任务 | 中 |

关键洞察：
Claude 默认在工艺和功能上已经不错了，
但在设计质量和原创性上，它往往产出平庸的"安全"方案。

所以 Anthropic 把权重向设计和原创倾斜，
并在评价标准中明确惩罚那些一眼就能认出的"AI 味"模式。
```

**评价标准本身就是提示词**:
```
一个有趣的发现是：
评价标准中的措辞本身就在塑造生成质量。

当标准中包含"最好的设计是博物馆级别的"这样的表述时，
生成器的输出会朝一个特定的视觉方向收敛。

换句话说，评价标准不只是"尺子"，
它本身就是一种隐性的"提示词"。
```

**创造性跳跃案例**:
```
Anthropic 让模型为一个荷兰艺术博物馆设计网站。

前九轮迭代中，模型一直在打磨一个暗色主题的着陆页——
已经很精致了，但还在"预期范围内"。

然后到了第十轮，它突然完全推翻了之前的方案，
重新设计成一个空间体验：
一个用 CSS 透视渲染的 3D 房间，
棋盘格地板，画作自由悬挂在墙上，
用户通过"门"在不同展厅之间导航——
不是滚动，不是点击，而是走进去。

这是 Anthropic 团队之前从未在单次生成中见过的创造性跳跃。
它发生在第十轮——
如果没有 Evaluator 的持续反馈推动模型"冒险"，
大概率会停留在第三四轮那个"还行"的方案上。

迭代不只是打磨，它还能产生质变。
```

### 28.9.19 Harness 的核心哲学与简化方法论

**核心哲学**:
```
"Harness 中的每个组件都编码了一个关于模型无法独立完成什么的假设，
而这些假设值得被压力测试。"

换句话说：
Harness 设计的艺术，
就是知道什么时候该加脚手架，什么时候该拆掉它。

当 Opus 4.6 发布后，它的长任务能力大幅提升，
原来需要 Sprint 拆分的任务，现在模型自己就能连续干两小时不跑偏。

于是 Anthropic 果断拆掉了 Sprint 构造，简化 Harness，
模型跑得更快，效果一样好。

这就是 Harness Design 的迭代哲学：
随着模型能力提升，不断移除不再承重的组件，
同时添加新的组件来突破新的能力天花板。
```

**简化方法论：控制变量法**:
```
Anthropic 在简化 Harness 时踩了一个坑——

第一次尝试时，他们"大刀阔斧"地砍掉了好几个组件，
同时还加了几个新想法。

结果既没能复现原来的效果，
也搞不清楚到底是哪个改动导致了退化。

后来他们换了一个笨办法：
每次只移除一个组件，跑一遍完整测试，看效果变化。

这就是控制变量法——中学物理教的东西，
但在 AI 工程中一样管用。

发现：
- Sprint 构造在 Opus 4.6 上已经不再承重 → 果断拆掉
- Planner 和 Evaluator 仍然必要 → 保留

关键洞察：
Evaluator 不是"要不要"的二元选择，
而是"当前任务是否超过了模型独自能做好的范围"的连续判断。

简单任务不需要，复杂任务必须有。
而"简单"和"复杂"的分界线，会随着模型升级而移动。
```

### 28.9.20 控制论视角：一切都是反馈回路

**实战案例：用一句话生成一个 DAW**:
```
任务：
"Build a fully featured DAW in the browser using the Web Audio API"
用一句话提示，让 AI 自主构建一个浏览器端的数字音频工作站。

成本拆解:

| Agent & Phase      | 耗时        | 成本     |
|-------------------|------------|---------|
| Planner           | 4.7 分钟    | $0.46   |
| Build (Round 1)   | 2 小时 7 分  | $71.08  |
| QA (Round 1)      | 8.8 分钟    | $3.24   |
| Build (Round 2)   | 1 小时 2 分  | $36.89  |
| QA (Round 2)      | 6.8 分钟    | $3.09   |
| Build (Round 3)   | 10.9 分钟   | $5.88   |
| QA (Round 3)      | 9.6 分钟    | $4.06   |
| **合计**          | **3 小时 50 分** | **$124.70** |

关键发现:
- Planner 只花了不到 5 分钟和 0.46 美元——成本极低但价值极高
- Build 是大头，第一轮就连续工作了 2 小时 7 分钟
- QA（Evaluator）每轮不到 10 分钟，成本只有几美元
- 但它发现了关键问题：
  - "音频录制只有按钮但没有实际麦克风捕获"
  - "片段无法在时间线上拖拽移动"
  - "效果器只有数字滑块没有图形化 EQ 曲线"
- 这些都是 Generator 自己不会承认的"半成品"
- $3 的 QA 捞回了 $70 的 Build 差点交付的半成品
- 这就是 Evaluator 的 ROI
```

**经典闭环控制系统**:
```
┌────────────────┐
│   目标（r）    │ ← 设定值（期望输出）
└───────┬────────┘
        │
        ▼
┌────────┐    ┌────────┐
│控制器 (C)│───▶│被控对象│───▶ y（实际输出）
└────────┘    │  (P)   │
      ▲       └────────┘
      │            │
      │       ┌────────┐
      └───────│传感器 (H)│
              └────────┘
           反馈信号 (y_m)
```

**控制论概念与 Agent 对应**:

| 控制论概念 | Agent 对应 | 具体例子 |
|-----------|-----------|---------|
| **目标 (r)** | 任务描述/产品规格 | "超过 jrsonnet 的跑分" |
| **控制器 (C)** | Planner + Generator | 制定计划、编写代码 |
| **被控对象 (P)** | 代码库/应用 | sjsonnet 源码 |
| **传感器 (H)** | Evaluator / 跑分工具 | ./mill bench.runRegressions |
| **误差 (e)** | 当前结果与目标的差距 | "还差 15% 性能" |
| **控制输入 (u)** | 代码变更 | 具体的优化 patch |

**核心洞察**:
```
所有的 Harness Design，本质上都在回答一个问题：
如何设计更好的反馈回路。
```

**PID 控制的类比**:
```
PID 控制器三个分量:
- P（比例）：当前误差有多大？→ 对应 Evaluator 的即时评分
- I（积分）：历史误差的累积？→ 对应 Memory / 经验积累
- D（微分）：误差的变化趋势？→ 对应"分数在改善还是停滞？"

Agent PID := P(即时评价) + I(经验记忆) + D(趋势感知)

Anthropic 的 Evaluator：主要是 P 分量
AutoResearch：兼具 P 和 D
OpenClaw 的记忆系统：引入了 I 分量

稳定性和超调：
过度控制和失控一样危险。

在 Agent 设计中：
- Evaluator 太严格 → Generator 大幅改动，最终产出不稳定
- Ralph 循环太频繁 → 系统在"做"和"检查"之间来回跳，效率极低
- 记忆更新太激进 → Agent 的"性格"快速漂移

好的 Harness Design 就是好的控制器调参：
足够的反馈来纠偏，又不至于过度反应导致震荡。
```

### 28.9.21 工程师的行动清单

**AutoResearch——"永不停止"的循环**:
```
Karpathy 的"让它跑一夜":

while True:  # 永不停止
    查看 git 状态
    想一个实验改进的主意
    修改 train.py
    提交代码
    运行实验
    读取结果
    if 验证损失改善了:
        保留这次提交
    else:
        git reset，回滚
    记录实验结果
    # 不要暂停问人类！人类可能在睡觉！

以每个实验约 5 分钟计算，一小时 12 个，一夜大约 100 个实验。
用户早上醒来，看到一堆实验结果，全部自主完成。

为什么 AutoResearch 不需要 Ralph Loop？
因为它的进度是可度量的。
验证损失要么下降了，要么没有——你骗不了数学。

任务类型与控制结构:

| 任务类型       | 可验证性          | 需要的控制结构         |
|--------------|-----------------|---------------------|
| ML 训练优化    | 高（loss 数值）   | 简单循环就够           |
| 代码修 bug     | 中（测试通过/失败） | 需要独立评价           |
| 前端设计       | 低（主观审美）     | 需要 GAN 式生成 - 评价循环 |
| 产品规划       | 极低             | 需要人类在环           |
```

**入门级：5 分钟上手**:
```
1. 给你的 AI 任务加一个反馈循环
   - 不要一次性让 AI 生成最终结果
   - 让它先做一个版本 → 自己评估 → 根据评估改进 → 重复

2. 设定明确的终止条件
   - 不要让 AI 自己决定"完成了没有"
   - 给它一个客观标准："通过这 5 个测试用例"或"性能提升 10%"

3. 记录失败原因
   - 每次 AI 的尝试失败了，要求它记录为什么失败
   - 下次不走同样的弯路
```

**进阶级：搭建你自己的 Harness**:
```
1. 分离生成和评价
   - 如果你的任务有主观成分（设计、文案、架构决策）
   - 一定要用独立的 Agent 来评估
   - 不要让同一个 Agent 既做又评

2. 设计结构化的上下文移交
   - 如果任务需要多个 session
   - 不要依赖对话历史的自然延续
   - 写一个明确的"移交文件"，包含当前进度、已完成项、待办项、关键决策

3. 建立"安全检查点"
   - 每个有效的中间结果都应该被保存（git commit、文件快照、数据库备份）
   - 这样即使后续出错，也能回滚到已知好的状态
```

**高级：构建自进化系统**:
```
1. 实现跨会话记忆
   - 让 Agent 在每次任务结束后提取经验
   - 存到持久化存储中
   - 下次类似任务，自动加载相关经验

2. 设计 Skill 生成器
   - 当 Agent 在某类任务上积累了足够多的经验
   - 自动把这些经验编译成可复用的 Skill 文件

3. 接入协作工具
   - 把 Agent 的工作状态推送到你团队使用的工具（钉钉、Slack、飞书）
   - 让 AI 的工作对团队可见、可追踪、可干预
```

### 28.9.22 后记：河道与水

> **模型就是水。**
> 它的能力——推理、生成、搜索——是物理性质决定的，
> 你改变不了 H2O 的分子结构。
>
> 但通过设计河道（Harness），
> 你可以让它灌溉农田，也可以让它驱动涡轮发电。
>
> **Harness Design 的精髓不是"限制"AI，而是"引导"它**——
> 让它的能力在正确的方向上充分释放。
>
> 一条好的河道不会让水变多，
> 但能让每一滴水都到达该到的地方。
>
> 模型的能力是上游给的，
> 但河道的设计是你的。
>
> 这是一个属于工程师的时代，
> 把我们从繁琐的 coding 中解放出来了。
> 你可能不是老板，但是你缺是你的几个 agents 的老板，
> claude、gemini、qwen、gpt 都得听你的。
>
> 我们不需要等下一代模型来解决所有问题——
> 通过更好的 Harness 设计，
> 今天的模型就能做到远超预期的事情。
