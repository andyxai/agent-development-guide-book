## 本章导读

传统 RAG 系统采用被动式固定流程（检索→生成），存在单次检索、缺乏反思、工具单一等局限性。本章深入讲解四种高级 RAG 技术：Agentic RAG 将检索能力赋予 Agent 实现自主决策；Self-RAG 通过反思令牌使模型自主评估检索和生成质量；CRAG 引入轻量级检索评估器并结合网络搜索扩展；RAG-Fusion 通过多查询生成和 RRF 融合提升检索覆盖率。本章将帮助你理解这些技术的核心原理、适用场景和组合策略，为实际项目中的 RAG 系统优化提供技术选型依据。

**学习目标**：
- 理解 Agentic RAG 的自主检索决策、多工具协同、迭代检索和自我修正机制

- 理解 CRAG 的检索评估器和修正策略
- 掌握 RAG-Fusion 的多查询生成和 RRF 融合算法
- 能够根据业务场景选择合适的高级 RAG 技术或组合方案

## 12.1 Agentic RAG - 自主检索决策

**总**：Agentic RAG 将 RAG 作为 Agent 的工具之一，使系统能够自主决定何时检索、如何检索、何时停止检索，实现从「被动响应」到「主动决策」的演进。

### 12.1.1 Agentic RAG 概念与演进

**技术起源**

Agentic RAG 是 RAG 与 Agent 技术融合的产物，最早出现在 2023-2024 年的 LangChain Agents 和 LlamaIndex Agents 实践中。其核心思想是将检索能力赋予 Agent，使 Agent 能够像人类一样自主决策检索行为。

- **提出年份**: 2023-2024
- **核心来源**: 
  - [LangChain Agents](https://python.langchain.com/docs/modules/agents/) (2023)
  - [LlamaIndex Agents](https://docs.llamaindex.ai/en/stable/module_guides/deploying/agents/) (2023)
  - [ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629) (ICLR 2023, arXiv:2210.03629)

**为什么需要 Agentic RAG？**

传统 RAG 的局限性：

1. **固定流程**: 检索 → 生成，无法根据中间结果调整策略
   - 例：检索结果质量差时，仍会基于低质量结果生成回答

2. **单次检索**: 只检索一次，无法进行多轮迭代
   - 例：第一次检索未找到答案时，不会主动调整查询重新检索

3. **工具单一**: 只有检索能力，无法调用其他工具辅助
   - 例：无法在检索后进行网络搜索验证或数学计算

4. **缺乏反思**: 不评估检索质量，不检测幻觉
   - 例：检索内容与问题不相关时，仍会生成看似合理但实际错误的回答

**Agentic RAG 的核心能力**

1. **自主检索决策**: 决定何时检索、何时不检索
2. **多工具协同**: 检索 + 搜索 + 计算 + 代码执行多工具组合
3. **迭代检索**: 基于中间结果进行多轮检索
4. **自我修正**: 检测检索失败并调整策略

**技术演进路线**

```
2023 Q1: ReAct (Reasoning + Acting)
  ↓
2023 Q3: Self-RAG (自我反思式 RAG)
  ↓
2024 Q1: CRAG (检索质量评估 + 网络搜索扩展)
  ↓
2024 Q2: RA-DIT (检索增强 + 指令微调)
  ↓
2024 Q4: Agentic RAG (多工具协同 + 迭代检索)
```

**案例应用**：漫剧设定一致性检查中，Agentic RAG 可以：
1. 先检索设定库，检查「轩辕墨的炎龙血脉」是否有明确定义
2. 如检索结果模糊，调用网络搜索查找类似玄幻设定作为参考
3. 如发现有冲突设定（如第 3 章说「火属性」，第 10 章说「金属性」），标记冲突并请求人工确认
4. 最终生成一致性报告，标注哪些设定已验证、哪些存疑

**常见误区**：认为「Agentic RAG 就是 RAG + Agent 简单叠加」。实际核心在于自主决策能力，Agent 需要判断何时检索、何时停止、何时切换工具。

### 12.1.2 自主检索决策：何时/如何检索

Agentic RAG 的核心是自主决策能力。Agent 需要判断当前任务是否需要检索、何时检索、以及如何检索。

**决策时机**

Agent 在以下场景应触发检索：

1. **知识密集型问题**
   - 问题涉及特定领域知识（如「轩辕墨的炎龙血脉有什么能力」）
   - 问题需要事实性信息（如「轩辕世家有几个主要成员」）

2. **不确定检测**
   - LLM 对回答置信度低（可通过概率分布或自我评估判断）
   - LLM 明确表示「我不确定」「我需要查证」

3. **事实核查**
   - 生成内容涉及关键事实（如剧情设定、角色关系）
   - 需要引用来源支撑（如学术论文、法律条文）

**决策不检索的场景**：

1. **通用知识**
   - 问题涉及常识（如「1+1 等于几」）
   - 问题涉及 LLM 训练数据中的通用知识（如「巴黎是法国首都」）

2. **创意生成**
   - 任务是创意写作（如「写一首关于春天的诗」）
   - 检索可能限制创造力

3. **对话延续**
   - 问题是多轮对话的延续（如「那后来呢」）
   - 上下文已包含足够信息

**决策实现方式**

**1. 基于规则的决策**

```python
def should_retrieve(query, context):
    # 规则 1: 包含特定关键词则检索
    if any(kw in query for kw in ['设定', '角色', '剧情', '世界观']):
        return True
    
    # 规则 2: 上下文不足则检索
    if len(context) < 500:  # 上下文少于 500 token
        return True
    
    # 规则 3: 问题类型判断
    if is_factual_question(query):  # 事实性问题
        return True
    
    return False
```

**2. 基于 LLM 的决策**

```python
decision_prompt = """
判断是否需要检索外部知识库来回答这个问题：

问题：{query}
当前上下文：{context}

请回答「需要检索」或「不需要检索」，并说明理由。
"""
```

**3. 基于置信度的决策**

```python
# 获取 LLM 对回答的置信度
response = llm.generate(query, return_logprobs=True)
confidence = calculate_confidence(response.logprobs)

if confidence < 0.7:  # 置信度低于阈值
    retrieve_and_regenerate()
```

**检索策略选择**

决定检索后，Agent 还需选择检索策略：

| 策略 | 适用场景 | 示例 |
|------|---------|------|
| **单次检索** | 简单事实查询 | 「轩辕墨是谁」 |
| **多轮检索** | 复杂问题、需要多步推理 | 「轩辕墨的盟友的敌人是谁」 |
| **混合检索** | 需要语义 + 精确匹配 | 「炎龙血脉的视觉效果描述」 |
| **图检索** | 需要关系推理 | 「轩辕墨的社交网络」 |

**案例应用**：漫剧剧本生成中，Agent 决策流程：
1. 收到任务：「生成第 11 章，轩辕墨与魔尊首次对决」
2. 判断：需要检索设定（角色能力、世界观规则）
3. 决策：使用混合检索（向量 + 图谱）
4. 检索：检索「轩辕墨」「魔尊」「炎龙血脉」「魔尊能力」
5. 验证：检查检索结果是否完整（如缺少魔尊能力，则补充检索）
6. 生成：基于检索结果生成剧本

**常见误区**：认为「检索越多越好」。实际过度检索会增加延迟和成本，且可能引入噪声。Agent 需要平衡检索收益与成本。

### 12.1.3 多工具协同：检索 + 搜索 + 计算

Agentic RAG 的强大之处在于能够协同多个工具，形成完整的问题解决链路。

**常见工具组合**

**1. RAG + 网络搜索**

- **RAG**: 检索内部知识库（如漫剧设定库）
- **网络搜索**: 补充外部信息（如类似作品参考、读者反馈）
- **协同方式**: 先 RAG 检索内部知识，如结果不足则网络搜索补充

**案例**：漫剧设定冲突检测
```
1. RAG 检索：检查内部设定库是否有冲突
2. 发现冲突：第 3 章「火属性」vs 第 10 章「金属性」
3. 网络搜索：查找类似玄幻作品的设定惯例
4. 综合判断：建议统一为「火属性」（更符合炎龙血脉设定）
```

**2. RAG + 代码执行**

- **RAG**: 检索问题相关信息
- **代码执行**: 进行数学计算、数据处理
- **协同方式**: RAG 提供数据，代码执行计算

**案例**：漫剧数据分析
```
1. RAG 检索：获取各章阅读量、评论数
2. 代码执行：计算趋势、相关性分析
3. 生成报告：「第 5 章轩辕墨觉醒剧情后，阅读量提升 35%」
```

**3. RAG + 计算器/单位转换**

- **RAG**: 检索数值信息
- **计算器**: 进行精确计算
- **协同方式**: RAG 提取数值，计算器验证计算

**案例**：玄幻世界观数值平衡
```
1. RAG 检索：「炎龙血脉灵力消耗设定」
2. 提取数值：「每次使用消耗 100 灵力，上限 1000」
3. 计算器验证：「连续使用 10 次后灵力耗尽」
4. 检查冲突：剧本中是否出现「连续使用 15 次」的剧情
```

**4. RAG + 数据库查询**

- **RAG**: 检索非结构化文档
- **数据库查询**: 检索结构化数据
- **协同方式**: RAG 处理文档，SQL 查询处理结构化数据

**案例**：漫剧项目管理
```
1. RAG 检索：剧本设定文档
2. SQL 查询：项目进度数据库（已完成章节、待审核章节）
3. 综合报告：「第 1-10 章已完成，第 11 章审核中，设定一致性检查通过」
```

**工具调度策略**

**1. 顺序执行**

工具按固定顺序执行，前一个工具的输出作为后一个工具的输入：
```
RAG 检索 → 提取数值 → 计算器验证 → 生成报告
```

**2. 并行执行**

独立工具并行执行，结果融合：
```
RAG 检索 (内部知识) ──┐
                      ├──→ 结果融合 → 生成回答
网络搜索 (外部知识) ──┘
```

**3. 条件分支**

根据中间结果决定下一步工具：
```
RAG 检索
  ↓
检索质量评估
  ├─ 高质量 → 直接生成
  ├─ 中质量 → 网络搜索补充
  └─ 低质量 → 重新检索或转人工
```

**工具选择决策**

Agent 使用 LLM 决定工具选择：
```python
tool_selection_prompt = """
你需要回答这个问题：{query}

可用工具：
1. knowledge_base: 检索内部知识库（漫剧设定、剧情文档）
2. web_search: 网络搜索（补充外部信息）
3. calculator: 数学计算
4. code_interpreter: 代码执行（数据分析、复杂计算）

请选择合适的工具序列，并说明理由。

示例输出：
工具序列：[knowledge_base, calculator]
理由：先检索设定文档获取数值，再用计算器验证灵力消耗计算。
"""
```

**案例应用**：漫剧剧本生成 Agent 的工具协同：
1. **RAG 检索**: 获取角色设定、世界观规则
2. **图谱查询**: 检查角色关系（确保不出现「盟友突然变敌人」的冲突）
3. **代码执行**: 统计前 10 章各角色出场次数（确保主角戏份充足）
4. **网络搜索**: 查找类似题材的热门剧情套路（作为参考）
5. **生成剧本**: 综合所有信息生成第 11 章

**常见误区**：认为「工具越多越好」。实际过多工具会增加决策复杂度和延迟。建议根据场景选择 3-5 个核心工具。

### 12.1.4 迭代检索与自我修正

**为什么需要迭代检索？**

1. **信息不足**: 第一次检索可能未找到完整答案
2. **查询模糊**: 初始查询不够精确，需要基于中间结果优化
3. **多步推理**: 复杂问题需要分步检索

**迭代检索流程**

```
第 1 轮：初始查询 → 检索 → 评估结果
  ↓
结果不足？
  ↓ 是
第 2 轮：优化查询（基于第 1 轮结果）→ 检索 → 评估结果
  ↓
结果不足？
  ↓ 是
第 3 轮：进一步细化查询 → 检索 → 评估结果
  ↓
结果充足 → 生成最终回答
```

**查询优化策略**

**1. 查询扩展**

基于第 1 轮结果中的关键词扩展查询：
```
第 1 轮查询：「轩辕墨的能力」
第 1 轮结果：提到「炎龙血脉」「轩辕剑法」「灵力感知」
第 2 轮查询：「炎龙血脉 详细设定」「轩辕剑法 招式」「灵力感知 限制」
```

**2. 查询细化**

将宽泛查询拆分为具体子查询：
```
初始查询：「轩辕世家的势力范围」
拆解为：
  - 「轩辕世家 地理位置」
  - 「轩辕世家 盟友势力」
  - 「轩辕世家 敌对势力」
```

**3. 查询修正**

根据第 1 轮结果的反馈修正查询：
```
第 1 轮查询：「轩辕墨的师傅」
第 1 轮结果：未找到直接提及「师傅」的内容
第 2 轮查询：「轩辕墨 师徒」「轩辕墨 指导」「轩辕墨 学习」
```

**迭代终止条件**

1. **结果充足**: 检索到的信息足够回答问题
2. **达到上限**: 达到最大迭代次数（通常 3-5 次）
3. **收益递减**: 新增信息量低于阈值（如<10%）
4. **超时**: 达到时间限制

**实践参数**：
- **最大迭代次数**: 3-5 次（超过后收益递减）
- **终止阈值**: 新增信息量<10% 或 相似度提升<0.05
- **超时限制**: 10-30 秒（根据场景调整）

**检索失败检测**

**1. 相似度阈值检测**

```python
# 检索结果与查询的相似度低于阈值
if max_similarity < 0.4:  # 阈值根据嵌入模型调整
    mark_as_failed("检索结果不相关")
    trigger_retry()
```

**2. LLM 自我评估**

```python
eval_prompt = """
评估以下检索结果是否足以回答问题：

问题：{query}
检索结果：{retrieved_chunks}

请评分（1-5 分）并说明理由：
- 5 分：完全足够，可直接回答
- 3 分：部分足够，需要补充
- 1 分：完全不足，需要重新检索
"""
```

**3. 答案可验证性检测**

```python
# 检查生成答案是否有引用支撑
if not has_citations(answer, retrieved_chunks):
    mark_as_hallucination("答案无引用支撑")
    trigger_regenerate()
```

**修正策略**

**1. 重新检索（Retry）**

- **调整查询**: 换一种表述方式
- **调整参数**: 增加 Top-K、调整相似度阈值
- **切换检索源**: 从向量检索切换到图谱检索

**2. 扩展检索（Expand）**

- **增加检索源**: 添加网络搜索、数据库查询
- **扩展查询**: 使用同义词、相关词扩展
- **多查询融合**: 生成多个变体查询，融合结果

**3. 降级策略（Degrade）**

- **标注不确定性**: 「根据有限信息，可能是...」
- **转人工处理**: 「设定库中没有明确记录，请人工确认」
- **使用通用知识**: 「基于通用玄幻设定，通常是...」（但标注非官方设定）

**修正流程**

```
检索 → 评估质量
  ↓
质量低？
  ↓ 是
选择修正策略
  ├─ 重新检索（调整查询/参数）
  ├─ 扩展检索（添加检索源）
  └─ 降级策略（标注不确定性/转人工）
  ↓
再次评估 → 质量合格？→ 生成回答
```

**案例应用**：漫剧设定检索的自我修正：
```
第 1 次检索：「轩辕墨的炎龙血脉」
  → 结果：相似度 0.35（低于阈值 0.4）
  → 检测：检索失败
  
修正策略：重新检索（调整查询）
第 2 次检索：「炎龙血脉 设定」「轩辕墨 能力 详细描述」
  → 结果：相似度 0.65（合格）
  → 生成：基于第 2 次检索结果回答
```

**常见误区**：认为「迭代次数越多答案越准确」。实际超过 3-5 次后收益急剧递减，且延迟和成本线性增长。认为「自我修正会大幅增加延迟」。实际合理设计的修正流程（如限制最大修正次数为 2 次）只会增加约 20-30% 的延迟，但能显著提升准确率（从 70% 提升至 90%+）。

## 12.2 Self-RAG - 自我反思式 RAG

**总**：Self-RAG 通过引入反思令牌（Reflection Tokens），使模型能够自主决定何时检索、如何生成、以及生成质量如何，是反思式 RAG 的基础范式。

### 12.2.1 Self-RAG 核心机制

**技术概述**

Self-RAG（Self-Reflective Retrieval-Augmented Generation）由 Asai 等人于 2023 年提出，是反思式 RAG 的基础范式。其核心创新是通过引入特殊的「反思令牌」（Reflection Tokens），使模型能够自主决定何时检索、如何生成、以及生成质量如何。

- **提出年份**: 2023
- **论文**: [Asai et al., Self-RAG: Learning to Retrieve, Generate, and Critique through Self-Reflection](https://arxiv.org/abs/2310.11511) (arXiv:2310.11511)
- **发表 venue**: 2024 年 ICLR（International Conference on Learning Representations）

**核心思想**

传统 RAG 的问题：
1. **被动检索**: 固定流程（检索→生成），无法根据任务需求调整
2. **缺乏反思**: 不评估检索质量和生成质量
3. **检索依赖**: 即使不需要检索（如通用知识问题）也会强制检索

Self-RAG 的解决方案：
1. **自主检索决策**: 模型自主决定是否需要检索
2. **自我批评**: 生成后评估答案的相关性和准确性
3. **迭代优化**: 基于批评反馈重新检索或重新生成

**反思令牌（Reflection Tokens）**

Self-RAG 引入五种特殊令牌，在生成过程中预测这些令牌：

| 令牌类型 | 令牌值 | 含义 |
|---------|--------|------|
| **检索令牌** | Retrieve/No Retrieve | 是否需要检索 |
| **相关令牌** | Relevant/Irrelevant | 检索结果是否相关 |
| **支持令牌** | Fully Supported/Partially Supported/No Support | 答案是否有检索结果支撑 |
| **有用令牌** | Useful/Not Useful | 答案是否有用 |
| **整体令牌** | 5 分制评分 | 整体质量评分 |

**生成流程**

```
输入问题 → 预测 [检索令牌]
  ↓
Retrieve: 检索文档 → 预测 [相关令牌]
  ↓
Relevant: 生成答案 → 预测 [支持令牌][有用令牌]
  ↓
Fully Supported + Useful: 输出答案
  ↓
Partially/No Support: 重新检索或重新生成
```

---

### 12.2.2 训练方法与实现

Self-RAG 通过两阶段训练实现反思能力：

**阶段 1：批评器训练（Critic Training）**
- 使用人工标注数据训练批评器
- 标注内容：检索结果相关性、答案支持度、答案有用性
- 数据量：约 10K 标注样本

**阶段 2：生成器训练（Generator Training）**
- 使用强化学习（PPO）训练生成器
- 奖励信号：批评器的评分
- 目标：最大化生成答案的批评评分

**实践参数**：
- **检索阈值**: 模型预测 Retrieve 概率>0.5 时触发检索
- **相关阈值**: 检索结果相关概率>0.6 才使用
- **支持阈值**: 答案完全支持概率<0.4 时触发重新生成
- **最大迭代**: 3 次（避免无限循环）

**实现要点**

**1. 反思令牌预测**
```python
# 伪代码示例
def self_rag_generate(question, model, retriever):
    # 步骤 1: 预测是否需要检索
    retrieve_token = model.predict_retrieve_token(question)
    
    if retrieve_token == "No Retrieve":
        # 不检索，直接生成
        return model.generate(question)
    
    # 步骤 2: 检索文档
    docs = retriever.retrieve(question)
    
    # 步骤 3: 预测检索结果相关性
    relevance_token = model.predict_relevance_token(question, docs)
    
    if relevance_token == "Irrelevant":
        # 检索结果不相关，重新检索或不检索
        return model.generate(question)
    
    # 步骤 4: 生成答案
    answer = model.generate(question, docs)
    
    # 步骤 5: 预测答案支持度
    support_token = model.predict_support_token(answer, docs)
    
    if support_token in ["Partially Supported", "No Support"]:
        # 答案支撑不足，重新生成
        return self_rag_generate(question, model, retriever, max_iter=2)
    
    return answer
```

**2. 强化学习训练**
```python
# 使用 PPO 训练生成器
from trl import PPOTrainer

# 奖励函数：基于批评器评分
def reward_function(answer, critic_score):
    return critic_score  # 5 分制评分

# PPO 训练循环
for batch in dataloader:
    responses = model.generate(batch.questions)
    rewards = [reward_function(r, s) for r, s in zip(responses, batch.critic_scores)]
    ppo_trainer.step(queries=batch, responses=responses, rewards=rewards)
```

**优势与局限**

**优势**：
1. **自主决策**: 减少不必要的检索（约 30-40% 的查询不需要检索）
2. **质量提升**: 通过自我批评，幻觉率降低 20-30%
3. **可解释性**: 反思令牌提供决策依据

**局限**：
1. **训练成本**: 需要标注数据和强化学习训练
2. **推理延迟**: 多次迭代增加延迟（约 1.5-2 倍）
3. **令牌开销**: 反思令牌增加 token 消耗（约 10-20%）

**案例应用**：漫剧设定一致性检查中，Self-RAG 可以：
1. 收到问题：「轩辕墨的炎龙血脉有什么限制？」
2. 预测检索令牌：Retrieve（需要检索设定库）
3. 检索后预测相关令牌：Relevant（找到相关设定）
4. 生成答案后预测支持令牌：Fully Supported（答案有设定文档支撑）
5. 预测有用令牌：Useful（答案完整有用）
6. 输出答案，附带反思令牌作为置信度标注

**常见误区**：认为「Self-RAG 只是加了几个令牌」。实际核心是通过强化学习让模型学会自我反思，令牌只是外在表现。

## 12.3 CRAG - 纠正式 RAG

**总**：CRAG 引入轻量级检索评估器，根据评估结果动态选择检索、网络搜索或模型内部知识，实现「评估-修正」循环。

### 12.3.1 CRAG 核心机制

**技术概述**

CRAG（Corrective Retrieval-Augmented Generation）由 Yan 等人于 2024 年提出，核心创新是引入轻量级检索评估器，根据评估结果动态选择检索、网络搜索或模型内部知识。

- **提出年份**: 2024
- **论文**: [Yan et al., CRAG: Corrective Retrieval-Augmented Generation](https://arxiv.org/abs/2401.15884) (arXiv:2401.15884)
- **发表 venue**: 2024 年 ACL（Association for Computational Linguistics）

**核心思想**

CRAG 的核心是「评估 - 修正」循环：
1. **评估检索质量**: 使用轻量评估器判断检索结果质量
2. **动态选择策略**: 根据评估结果选择不同处理策略
3. **网络搜索扩展**: 当内部检索不足时，使用网络搜索补充

**检索评估器（Retrieval Evaluator）**

评估器将检索结果分为三类：

| 评估结果 | 含义 | 置信度范围 |
|---------|------|-----------|
| **Correct（相关）** | 检索结果足够回答问题 | >0.7 |
| **Incorrect（不相关）** | 检索结果与问题无关 | <0.3 |
| **Ambiguous（不确定）** | 检索结果部分相关，但不完整 | 0.3-0.7 |

**评估器实现**：
- **模型**: 轻量 BERT 或 RoBERTa（约 100M 参数）
- **输入**: 问题 + 检索结果拼接
- **输出**: 三分类（Correct/Incorrect/Ambiguous）
- **训练数据**: 约 5K 人工标注样本（问题 - 检索结果 - 相关性标签）

---

### 12.3.2 修正策略与知识重组

**修正策略（Corrective Strategies）**

根据评估结果选择不同策略：

**1. Correct（相关）→ 直接使用**
```
检索结果 → 生成答案 → 输出
```

**2. Incorrect（不相关）→ 放弃检索**
```
放弃检索结果 → 使用模型内部知识 → 生成答案 → 输出
```
- 适用场景：通用知识问题（如「1+1 等于几」）
- 优势：避免低质量检索结果干扰

**3. Ambiguous（不确定）→ 网络搜索扩展**
```
检索结果不足 → 网络搜索补充 → 融合内部 + 外部结果 → 生成答案
```
- **搜索查询生成**: 基于原问题和已检索结果生成搜索查询
- **结果过滤**: 过滤低质量搜索结果（相似度<0.5）
- **结果融合**: 使用 RRF 融合内部检索和外部搜索结果

**知识分解与重组（Knowledge Decomposition & Recombination）**

CRAG 的创新技术，用于融合多源知识：

**1. 知识分解**
- 将检索结果拆分为独立的事实片段（Fact Snippets）
- 例：「轩辕墨是轩辕世家的少主，拥有炎龙血脉」→ 
  - 片段 1：「轩辕墨是轩辕世家的少主」
  - 片段 2：「轩辕墨拥有炎龙血脉」

**2. 知识重组**
- 根据问题相关性重新组合事实片段
- 过滤不相关片段，保留相关片段
- 生成连贯的答案

**实践参数**：
- **评估器阈值**: Correct>0.7, Incorrect<0.3, Ambiguous=0.3-0.7
- **网络搜索 Top-K**: 5-10 个搜索结果
- **融合权重**: 内部检索 0.6，外部搜索 0.4（可根据场景调整）
- **最大迭代**: 2 次（评估→修正→再评估）

**实现要点**

**1. 检索评估器**
```python
from transformers import AutoModelForSequenceClassification, AutoTokenizer

class RetrievalEvaluator:
    def __init__(self, model_name="bert-base-uncased"):
        self.tokenizer = AutoTokenizer.from_pretrained(model_name)
        self.model = AutoModelForSequenceClassification.from_pretrained(
            model_name, num_labels=3
        )
        self.model.load_state_dict(torch.load("crag_evaluator.pth"))
    
    def evaluate(self, question, retrieved_docs):
        # 拼接问题和文档
        input_text = f"Question: {question} Document: {retrieved_docs[0]}"
        inputs = self.tokenizer(input_text, return_tensors="pt", truncation=True, max_length=512)
        
        # 预测
        with torch.no_grad():
            outputs = self.model(**inputs)
            probs = torch.softmax(outputs.logits, dim=-1)
        
        # 三分类：Correct(0), Ambiguous(1), Incorrect(2)
        label = torch.argmax(probs, dim=-1).item()
        confidence = probs[0, label].item()
        
        labels = ["Correct", "Ambiguous", "Incorrect"]
        return labels[label], confidence
```

**2. 网络搜索扩展**
```python
import requests

def web_search(query, api_key, num_results=5):
    """调用 Bing/Google 搜索 API"""
    url = "https://api.bing.microsoft.com/v7.0/search"
    params = {"q": query, "count": num_results}
    headers = {"Ocp-Apim-Subscription-Key": api_key}
    
    response = requests.get(url, headers=headers, params=params)
    results = response.json()["webPages"]["value"]
    
    return [r["snippet"] for r in results]

def crag_generate(question, retriever, evaluator, web_search_api):
    # 步骤 1: 内部检索
    docs = retriever.retrieve(question)
    
    # 步骤 2: 评估检索质量
    label, confidence = evaluator.evaluate(question, docs)
    
    # 步骤 3: 根据评估结果选择策略
    if label == "Correct":
        # 直接使用内部检索结果
        return generate_answer(question, docs)
    
    elif label == "Incorrect":
        # 放弃检索，使用模型内部知识
        return generate_answer(question, [])
    
    else:  # Ambiguous
        # 网络搜索补充
        search_query = f"{question} 详细"
        web_results = web_search_api(search_query)
        
        # 融合内部和外部结果
        all_docs = docs + web_results
        return generate_answer(question, all_docs)
```

**优势与局限**

**优势**：
1. **轻量级**: 评估器仅 100M 参数，推理延迟<50ms
2. **灵活扩展**: 网络搜索补充内部检索不足
3. **训练简单**: 只需 5K 标注数据，微调成本低

**局限**：
1. **网络依赖**: 需要外部搜索 API（如 Bing、Google）
2. **噪声风险**: 网络搜索结果质量不稳定
3. **评估器泛化**: 跨领域性能可能下降

**案例应用**：漫剧设定检索中，CRAG 的工作流程：
```
问题：「轩辕墨的炎龙血脉有什么视觉效果？」

1. 内部检索：检索设定库
   → 找到：「炎龙血脉，火属性能力」
   
2. 评估器评估：Ambiguous（置信度 0.5）
   → 有相关信息，但不完整（缺少视觉效果描述）
   
3. 网络搜索扩展：
   → 搜索查询：「玄幻小说 炎龙血脉 视觉效果 描写」
   → 找到参考：「周身环绕火焰龙影，双眼呈金黄色」
   
4. 知识融合：
   → 内部：「炎龙血脉，火属性能力」
   → 外部：「周身环绕火焰龙影，双眼呈金黄色」
   → 融合生成：「炎龙血脉是火属性能力，发动时周身环绕火焰龙影，双眼呈金黄色」
```

**常见误区**：认为「CRAG 总是需要网络搜索」。实际只有 Ambiguous 场景才需要，Correct 场景直接使用内部检索（约 60-70% 的查询）。

## 12.4 RAG-Fusion - 融合式 RAG

**总**：RAG-Fusion 通过生成多个查询变体，分别检索后使用 RRF（Reciprocal Rank Fusion）融合结果，提升检索覆盖率和准确率。

### 12.4.1 RAG-Fusion 核心机制

**技术概述**

RAG-Fusion 由 Rackauckas 于 2024 年提出，核心思想是生成多个查询变体，分别检索后使用 RRF（Reciprocal Rank Fusion）融合结果，提升检索覆盖率和准确率。

- **提出年份**: 2024
- **论文**: [Rackauckas, RAG-Fusion: A New Take on Retrieval-Augmented Generation](https://arxiv.org/abs/2402.03367) (arXiv:2402.03367)
- **来源**: ArXiv preprint（社区广泛采用）

**核心思想**

传统 RAG 的问题：
1. **单查询局限**: 单一查询可能遗漏相关信息
2. **表述偏差**: 用户查询表述可能不够精确
3. **覆盖不足**: 单一检索结果覆盖范围有限

RAG-Fusion 的解决方案：
1. **多查询生成**: 使用 LLM 生成多个相关查询变体
2. **并行检索**: 对每个查询分别检索
3. **RRF 融合**: 使用倒数排名融合多源结果

**多查询生成（Multi-Query Generation）**

使用 LLM 生成查询变体：

```python
query_generation_prompt = """
请为以下问题生成 5 个相关的查询变体，覆盖不同角度：

原问题：{original_query}

示例：
原问题：「轩辕墨的炎龙血脉有什么能力？」
变体 1：「炎龙血脉 技能 列表」
变体 2：「轩辕墨 能力 详细介绍」
变体 3：「炎龙血脉 攻击 防御 辅助」
变体 4：「轩辕墨 战斗 招式」
变体 5：「炎龙血脉 特殊效果」

请生成 5 个变体：
"""
```

**生成的查询变体类型**：
1. **同义改写**: 换一种表述方式
2. **细化查询**: 拆分为具体子问题
3. **扩展查询**: 添加相关关键词
4. **抽象查询**: 提升到更高层次
5. **反向查询**: 从反面角度提问

---

### 12.4.2 RRF 融合算法

**RRF 融合（Reciprocal Rank Fusion）**

RRF 是经典的排名融合算法，无需调权重即可自动平衡不同检索源：

**公式**：
```
score(d) = Σ 1 / (k + rank_i(d))
```

- `score(d)`: 文档 d 的最终得分
- `rank_i(d)`: 文档 d 在第 i 个查询检索结果中的排名
- `k`: 平滑常数，通常设为 **60**（经验值）

**计算示例**：
```
查询 1 结果：[A(1), B(2), C(3)]
查询 2 结果：[B(1), C(2), D(3)]
查询 3 结果：[C(1), A(2), E(3)]

A 的得分：1/(60+1) + 1/(60+2) = 0.0164 + 0.0161 = 0.0325
B 的得分：1/(60+2) + 1/(60+1) = 0.0161 + 0.0164 = 0.0325
C 的得分：1/(60+3) + 1/(60+2) + 1/(60+1) = 0.0159 + 0.0161 + 0.0164 = 0.0484
D 的得分：1/(60+3) = 0.0159
E 的得分：1/(60+3) = 0.0159

最终排名：C(0.0484) > A(0.0325) = B(0.0325) > D(0.0159) = E(0.0159)
```

**k 值选择**：
- **k=60**: 经验值，平衡排名差异
- **k 较小**（如 20）: 排名靠前的文档得分差异更大
- **k 较大**（如 100）: 排名差异被过度平滑

**RAG-Fusion 流程**

```
1. 输入原问题
   ↓
2. LLM 生成 5 个查询变体
   ↓
3. 对每个变体分别检索（Top-K=10）
   ↓
4. 使用 RRF 融合 5 个结果列表（共 50 个文档）
   ↓
5. 取 Top-K（如 Top-10）作为最终检索结果
   ↓
6. 送入 LLM 生成答案
```

**实践参数**：
- **查询变体数量**: 4-6 个（太少覆盖不足，太多成本过高）
- **每查询 Top-K**: 10-20 个（融合后取 Top-10）
- **RRF k 值**: 60（经验值）
- **最终 Top-K**: 5-10 个（根据生成模型上下文长度调整）

**实现要点**

**1. 多查询生成**
```python
def generate_queries(original_query, llm, num_variants=5):
    prompt = f"""
请为以下问题生成 {num_variants} 个相关的查询变体，覆盖不同角度：

原问题：{original_query}

请生成变体（每行一个）：
"""
    response = llm.generate(prompt)
    queries = response.strip().split("\n")
    
    # 确保包含原问题
    queries = [original_query] + queries[:num_variants-1]
    
    return queries
```

**2. RRF 融合**
```python
def reciprocal_rank_fusion(results_lists, k=60):
    """
    results_lists: 多个查询的检索结果列表
    k: 平滑常数
    """
    # 计算每个文档的 RRF 得分
    scores = {}
    
    for results in results_lists:
        for rank, doc in enumerate(results):
            doc_id = doc["id"]
            if doc_id not in scores:
                scores[doc_id] = {"doc": doc, "score": 0}
            
            # RRF 公式：1 / (k + rank)
            scores[doc_id]["score"] += 1.0 / (k + rank + 1)
    
    # 按得分排序
    sorted_docs = sorted(scores.values(), key=lambda x: x["score"], reverse=True)
    
    return [item["doc"] for item in sorted_docs]

def rag_fusion_generate(question, retriever, llm):
    # 步骤 1: 生成多查询
    queries = generate_queries(question, llm, num_variants=5)
    
    # 步骤 2: 分别检索
    results_lists = []
    for query in queries:
        results = retriever.retrieve(query, top_k=10)
        results_lists.append(results)
    
    # 步骤 3: RRF 融合
    fused_results = reciprocal_rank_fusion(results_lists, k=60)
    
    # 步骤 4: 取 Top-10 生成答案
    top_results = fused_results[:10]
    return generate_answer(question, top_results)
```

**优势与局限**

**优势**：
1. **覆盖率提升**: 多查询覆盖不同角度，召回率提升 15-25%
2. **无需调参**: RRF 自动平衡，无需手动设置权重
3. **实现简单**: 只需 LLM 查询生成 + 多次检索 + RRF 融合

**局限**：
1. **成本增加**: 多次检索增加延迟和 API 调用成本（约 3-5 倍）
2. **冗余结果**: 不同查询可能检索到相同文档
3. **查询质量依赖**: 生成的查询变体质量影响最终效果

**案例应用**：漫剧设定检索中，RAG-Fusion 的工作流程：
```
原问题：「轩辕墨的社交关系是怎样的？」

1. 生成查询变体：
   - 变体 1：「轩辕墨 盟友 朋友」
   - 变体 2：「轩辕墨 敌人 对手」
   - 变体 3：「轩辕墨 师尊 徒弟」
   - 变体 4：「轩辕墨 家族 亲人」
   - 变体 5：「轩辕墨 感情线 爱慕」

2. 分别检索（每个变体 Top-10）：
   - 变体 1 找到：李逍遥、林婉儿
   - 变体 2 找到：魔尊、血煞老祖
   - 变体 3 找到：云逸真人
   - 变体 4 找到：轩辕烈（父）、柳如烟（母）
   - 变体 5 找到：林婉儿（爱慕）

3. RRF 融合：
   - 林婉儿在变体 1 和变体 5 都出现，得分更高
   - 融合后排名：林婉儿 > 李逍遥 > 魔尊 > 轩辕烈 > ...

4. 生成答案：
   - 盟友：李逍遥、林婉儿
   - 敌人：魔尊、血煞老祖
   - 师尊：云逸真人
   - 亲人：轩辕烈（父）、柳如烟（母）
   - 爱慕：林婉儿
```

**常见误区**：认为「查询变体越多越好」。实际超过 6 个后收益递减（召回率提升<5%），但成本线性增长。

## 12.5 技术对比与选型

**总**：四种高级 RAG 技术各有特点，Self-RAG 适合高准确性要求场景，CRAG 适合轻量级部署和快速落地，RAG-Fusion 适合召回率优先场景，Agentic RAG 适合复杂多工具协同场景。三者不是互斥的，可以组合使用形成更强的系统。

### 12.5.1 技术对比

**Self-RAG、CRAG、RAG-Fusion 对比**

| 特性 | Self-RAG | CRAG | RAG-Fusion |
|------|----------|------|------------|
| **核心创新** | 反思令牌自主决策 | 检索评估 + 网络扩展 | 多查询生成 + RRF 融合 |
| **检索决策** | 模型自主决定 | 评估器决定 | 固定多查询 |
| **修正策略** | 迭代重新生成 | 网络搜索补充 | 多查询融合 |
| **训练需求** | 需要强化学习微调 | 轻量微调（5K 数据） | 无需训练 |
| **推理延迟** | 高（1.5-2 倍） | 中（1.2-1.5 倍） | 高（3-5 倍检索） |
| **API 成本** | 中（+10-20% token） | 中（+网络搜索成本） | 高（3-5 倍检索） |
| **实现复杂度** | 高（需训练） | 中（需评估器） | 低（纯推理） |
| **幻觉抑制** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| **召回率提升** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |

**Agentic RAG 定位**

Agentic RAG 是更高层级的架构，可以整合 Self-RAG、CRAG、RAG-Fusion 作为其检索模块：

| 特性 | Agentic RAG | 其他三种技术 |
|------|-------------|-------------|
| **定位** | 系统架构层 | 检索优化层 |
| **核心能力** | 自主决策 + 多工具协同 | 检索质量提升 |
| **工具数量** | 3-5 个核心工具 | 单一检索工具 |
| **适用场景** | 复杂问题解决 | 检索优化 |

### 12.5.2 适用场景

**Self-RAG 适用场景**：
1. **高准确性要求**: 医疗、法律等不能容忍幻觉的场景
2. **检索成本高**: 需要减少不必要检索（节省 30-40% 检索）
3. **有训练资源**: 可以投入标注数据和计算资源微调

**CRAG 适用场景**：
1. **内部知识不足**: 需要网络搜索补充（如时事、最新研究）
2. **轻量级部署**: 评估器仅 100M 参数，适合边缘部署
3. **快速落地**: 只需 5K 数据微调，1-2 周可上线

**RAG-Fusion 适用场景**：
1. **召回率优先**: 需要尽可能找到相关信息（如研究调研）
2. **无需训练**: 没有标注数据或训练资源
3. **查询表述模糊**: 用户查询不够精确，需要多角度覆盖

**Agentic RAG 适用场景**：
1. **复杂问题解决**: 需要多工具协同（检索+搜索+计算）
2. **迭代推理**: 需要多轮检索和中间结果评估
3. **自主决策**: 需要根据任务类型动态选择策略

### 12.5.3 组合使用策略

三者不是互斥的，可以组合使用：

**策略 1：CRAG + RAG-Fusion**
```
1. 使用 RAG-Fusion 生成多查询并检索
2. 使用 CRAG 评估器评估融合结果质量
3. 如 Ambiguous，触发网络搜索补充
```

**策略 2：Self-RAG + CRAG**
```
1. 使用 Self-RAG 自主决定检索时机
2. 检索后使用 CRAG 评估器评估质量
3. 如质量低，触发 CRAG 修正策略
```

**策略 3：三者融合（最强）**
```
1. Self-RAG 决定是否需要检索
2. 需要检索时，RAG-Fusion 生成多查询
3. 检索结果用 CRAG 评估器评估
4. 如 Ambiguous，网络搜索补充
5. 生成答案后，Self-RAG 反思令牌评估质量
```

**策略 4：Agentic RAG + 反思式 RAG**
```
1. Agentic RAG 作为顶层架构，负责工具调度和决策
2. 检索任务委托给 RAG-Fusion（提升召回率）
3. 检索结果用 CRAG 评估器评估质量
4. 生成答案后用 Self-RAG 反思令牌评估
5. 如质量低，Agentic RAG 触发修正策略（重新检索/网络搜索/转人工）
```

**案例应用**：漫剧设定检索系统采用 CRAG + RAG-Fusion 组合：
- **RAG-Fusion**: 生成多查询覆盖角色设定、关系、剧情等多角度
- **CRAG**: 评估融合结果，如不完整则网络搜索参考类似作品
- **效果**: 召回率提升 25%，设定完整性从 75% 提升至 92%

**常见误区**：认为「必须三选一」。实际可以根据场景组合使用，如 RAG-Fusion 提升召回 + CRAG 评估质量 + Self-RAG 自主决策。

### 12.5.4 实践建议

1. **从小规模开始**: 先用 RAG-Fusion（无需训练）验证效果，再考虑 Self-RAG/CRAG
2. **监控指标**: 跟踪召回率、准确率、延迟、成本四个维度
3. **A/B 测试**: 与基线 RAG 对比，确保提升显著（召回率+10% 以上）
4. **成本控制**: 设置最大迭代次数和查询变体数量上限
5. **渐进式优化**: 先优化检索质量，再引入反思机制，最后考虑多工具协同

## 12.6 简单举例

### 案例设计
- **案例名称**：漫剧设定检索系统优化
- **涉及知识点**：Agentic RAG 自主决策、Self-RAG 反思令牌、CRAG 评估器、RAG-Fusion 多查询融合
- **案例目标**：帮助理解如何根据业务需求选择合适的高级 RAG 技术组合
- **案例内容要点**：
  * **场景描述**：漫剧创作团队需要构建设定检索系统，支持角色设定查询、剧情一致性检查、世界观规则验证。初始系统使用传统 RAG，召回率 65%，设定冲突率 15%。
  * **技术应用**：采用 CRAG + RAG-Fusion 组合。RAG-Fusion 生成 5 个查询变体覆盖不同角度（角色能力/关系/剧情/世界观），CRAG 评估器评估融合结果质量（Correct 直接使用、Ambiguous 网络搜索补充）。对高准确性要求的场景（如角色能力查询）引入 Self-RAG 反思令牌。

- **注意事项**：不展开 CRAG 评估器训练细节（见 12.3 节）和 RRF 算法推导（见 12.4 节）

## 本章总结

本章深入讲解了四种高级 RAG 技术：

1. **Agentic RAG**：将 RAG 作为 Agent 的工具，实现自主检索决策、多工具协同、迭代检索和自我修正。核心是从「被动响应」到「主动决策」的演进，适合复杂问题解决场景。

2. **Self-RAG**：通过反思令牌（Retrieve/Relevant/Supported/Useful）使模型自主评估检索和生成质量。核心是通过强化学习训练模型学会自我反思，幻觉率可降低 20-30%，但需要标注数据和训练资源。

3. **CRAG**：引入轻量级检索评估器（100M 参数），根据评估结果（Correct/Ambiguous/Incorrect）动态选择策略。核心是「评估-修正」循环和网络搜索扩展，训练简单（5K 数据），适合快速落地。

4. **RAG-Fusion**：通过多查询生成和 RRF 融合提升检索覆盖率。核心是 RRF 算法（score(d) = Σ 1 / (k + rank_i(d))），无需调参即可自动平衡不同检索源，召回率可提升 15-25%。

**技术选型建议**：
- **准确性优先**：Self-RAG 或 Self-RAG + CRAG 组合
- **召回率优先**：RAG-Fusion 或 RAG-Fusion + CRAG 组合
- **快速落地**：RAG-Fusion（无需训练）或 CRAG（轻量微调）
- **复杂场景**：Agentic RAG 整合反思式 RAG 技术

**实践路径**：从小规模开始（RAG-Fusion 验证效果）→ 监控指标（召回率/准确率/延迟/成本）→ 渐进式优化（先检索质量，再反思机制，最后多工具协同）。
