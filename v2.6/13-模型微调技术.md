## 13.1 模型微调概述

### 为什么需要微调

大模型虽然强大，但在特定领域或特定任务上可能存在不足。

**微调的价值**:
- ✅ 提升特定任务性能
- ✅ 注入领域知识
- ✅ 统一输出风格
- ✅ 增强安全性

### 微调技术分类

```
模型微调技术
├── SFT (Supervised Fine-Tuning)
│   ├── 全量微调
│   ├── 参数高效微调 (PEFT)
│   │   ├── LoRA
│   │   ├── QLoRA
│   │   └── Prefix Tuning
│   └── 指令微调
├── RLHF (Reinforcement Learning from Human Feedback)
│   ├── 奖励模型训练
│   ├── PPO 优化
│   └── DPO (Direct Preference Optimization)
└── 领域自适应
    ├── 继续预训练
    └── 领域微调
```

## 13.2 SFT (Supervised Fine-Tuning)

### 13.2.1 SFT 原理

使用标注数据对预训练模型进行有监督微调。

**基本流程**:
```
预训练模型 → 准备标注数据 → 监督微调 → 评估 → 部署
```

### 13.2.2 参数高效微调 (PEFT)

**LoRA (Low-Rank Adaptation)**:
- 原理：通过低秩矩阵更新权重
- 优势：参数量少、训练快、效果好
- 适用：资源有限场景

**QLoRA**:
- 4bit 量化基座模型
- 显存需求降低 4 倍
- 性能与 LoRA 相当

---

## 13.3 RLHF (Reinforcement Learning from Human Feedback)

### 13.3.1 RLHF 原理

通过人类反馈强化学习，使模型输出更符合人类偏好。

**三阶段流程**:
```
阶段 1: SFT 微调
  ↓
阶段 2: 奖励模型训练
  ↓
阶段 3: RL 优化 (PPO)
```

### 13.3.2 DPO (Direct Preference Optimization)

**DPO 优势**:
- 不需要单独的奖励模型
- 训练更稳定
- 计算成本更低

**DPO vs RLHF**:
| 维度 | RLHF | DPO |
|------|------|-----|
| 复杂度 | 高 (三阶段) | 低 (单阶段) |
| 稳定性 | 中 | 高 |
| 成本 | 高 | 低 |
| 效果 | 好 | 相当或更好 |

## 13.4 实战案例

### 13.4.1 客服 Agent 微调

**目标**: 微调模型成为专业客服

**数据准备**:
```python
data = []
for qa_pair in qa_pairs:
    data.append({
        "instruction": "作为客服，回答用户问题",
        "input": qa_pair["question"],
        "output": qa_pair["answer"]

```

**效果评估**:

**效果评估**:

## 13.5 最佳实践

### 13.5.1 数据质量优先

**原则**:
1. **质量 > 数量**: 1000 高质量样本 > 10000 低质量样本
2. **多样性**: 覆盖各种场景
3. **一致性**: 风格统一
4. **平衡性**: 避免偏见

### 13.5.2 避免灾难性遗忘

**方法**:
1. **混合训练**: 混合通用数据和任务数据
2. **正则化**: 使用 EWC 等技术
3. **LoRA**: 只训练少量参数
4. **评估**: 定期评估通用能力

## 13.7 总结

### 技术选型建议

| 场景 | 推荐技术 |
|------|---------|
| 资源充足 | 全量 SFT + RLHF |
| 资源有限 | LoRA/QLoRA |
| 快速实验 | DPO |
| 领域适配 | 继续预训练 + SFT |

### 注意事项

1. **数据质量**: 决定微调效果上限
2. **评估全面**: 避免过拟合
3. **成本控制**: 选择合适的技术
4. **持续迭代**: 持续收集数据优化
